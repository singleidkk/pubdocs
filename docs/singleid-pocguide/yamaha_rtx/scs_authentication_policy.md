# SCS評価制度におけるYAMAHA RTXのVPN認証・管理者認証強化の考え方

文書更新日: 2026-05-16

参照資料: IPA「SCS評価制度 ★3・★4 要求事項・評価基準」（2026-05-16参照）

本書は、YAMAHA RTX のリモートアクセスVPN利用者認証、および RTX への管理者ログイン認証に SingleID を利用する場合に、SCS評価制度の観点でどのように説明できるかを整理した文書です。

SCS評価制度への適合を単独で保証するものではありませんが、VPN利用者認証とネットワーク機器管理者認証の強化を説明する際の判断材料として利用することを目的としています。

!!! summary "この資料の要約"
    YAMAHA RTX と SingleID を組み合わせることで、リモートアクセスVPNの利用者認証に加えて、RTX管理者ログインも、ユーザ単位の認証、多要素認証、グループ単位のアクセス制御、認証ログ確認の観点で整理しやすくなります。

    これにより、SCS評価制度におけるユーザID管理、管理者ID管理、認証強度、多要素認証、アクセス権管理、ネットワーク境界防護、ログ管理の考え方に沿った対策として説明しやすくなります。

    ただし、RTX本体のファームウェア管理、フィルタ設定、VPN暗号設定、管理アクセス経路の制限、ローカル管理者アカウント、緊急時運用、ログ保管期間、申請・承認フローは、SingleIDだけでは完結しません。顧客側の運用ルールおよびRTX側設定と合わせて確認する必要があります。

## 想定読者

本書は、次の立場の方が、YAMAHA RTX と SingleID の構成をSCS評価制度の文脈で評価・説明するための資料です。

- セキュリティ監査、内部統制、リスク管理を担当する方
- CIO、CISO、情報システム部門の責任者
- 情報システム部門でネットワーク機器やリモートアクセスVPNの運用方針を検討する方
- 営業、プリセールス、SEとして顧客説明の裏付けを確認する方

本書は設定手順書ではありません。具体的な設定方法は、参考情報に記載している設定ガイドを確認してください。

!!! info
    本書は、2026-05-16時点でIPAが公開している「SCS評価制度 ★3・★4 要求事項・評価基準」を参照しています。SCS評価制度の解説書や要求事項・評価基準は今後更新される可能性があるため、顧客説明時には最新の公式資料を確認してください。

| RTX管理者ログイン | RTXへ管理者としてログインする際の認証。SingleIDによるパスワード認証、ワンタイムパスワード認証、ユーザ・グループ制御を扱います。 |

## SingleIDとRTXで分担する範囲

SCS評価制度の観点では、SingleIDの機能だけでなく、RTX側設定と顧客側の運用ルールを組み合わせて説明する必要があります。主な分担は次のとおりです。

| 項目 | SingleIDで説明しやすくなること | RTX側・顧客運用側で確認すること |
|---|---|---|
| VPN利用者認証 | ユーザ単位の認証、ワンタイムパスワード認証、グループ制御、認証ログ | VPN設定、暗号方式、到達可能範囲、ローカルVPNユーザの扱い |
| RTX管理者ログイン | 管理者ごとの認証、ワンタイムパスワード認証、許可ユーザ・許可グループ制御 | 管理画面の公開範囲、接続元制限、ローカル管理者の扱い |
| 権限管理 | SingleIDの管理アクセス認証設定による管理権限・一般権限の割り当て | RTX側の管理アクセス方式、接続元制限等の設定 |
| ログ確認 | 認証ログによる成功・失敗確認 | RTX側ログ、保存期間、監視頻度、インシデント時の確認手順 |

## SCS評価制度との関係

本書では、IPAが公開している「SCS評価制度 ★3・★4 要求事項・評価基準」（2026-05-16参照）のうち、YAMAHA RTX のVPN利用者認証および管理者認証と関係する項目を整理します。

SCS評価制度の公開要求事項では、現時点でリモートアクセスVPNそのものを名指しした要求は確認できません。ただし、認証の強度、多要素認証、パスワード設定ルール、ネットワーク境界防護、ログ取得に関する項目は、VPN利用者認証やRTX管理者認証を評価する際の判断材料になります。

本構成は、SCS評価制度への適合を単独で保証するものではありません。SCS評価制度では、認証方式だけでなく、資産管理、ログ管理、ネットワーク境界防護、運用ルールなども含めて評価されます。そのため、本構成は「SCS評価制度の認証強度や多要素認証の考え方に沿った対策」として位置づけるのが適切です。

YAMAHA RTX と SingleID を組み合わせることで、リモートアクセスVPN利用者認証とRTX管理者認証を、ユーザ単位の認証、多要素認証、グループ単位のアクセス制御、認証ログ確認の観点で整理しやすくなります。

特に、リモートアクセスVPNはインターネット経由で社内環境へ接続する入口になるため、パスワードのみで認証するのではなく、パスワードとワンタイムパスワードを組み合わせることにより、認証強度を高める構成として説明できます。また、RTX管理者ログインにもSingleIDによる認証とワンタイムパスワードを組み合わせることで、ネットワーク機器管理者認証の統制を説明しやすくなります。

一方で、SCS評価制度は認証方式だけで評価されるものではありません。SCS評価制度の観点で説明する場合は、以下も合わせて確認してください。

- SingleID側のパスワードポリシー
- ワンタイムパスワードの利用対象ユーザ
- VPN利用者認証とRTX管理者認証のどちらにSingleIDを適用するか
- SingleID側の管理アクセス認証設定と、管理権限・一般権限の割り当て
- YAMAHA RTX側のVPN設定、管理アクセス方式、接続元制限
- RTXのローカル管理者アカウント、緊急時アカウント、管理パスワードの扱い
- VPN装置と認証基盤間の通信経路制限
- 認証ログ、VPN接続ログ、RTX管理ログの取得・保管
- 申請・承認フロー、棚卸、ログ監視などの顧客側運用

関連する要求事項は、次の観点で整理できます。

| 分類 | 主な関連項目 | 主な対象 | 本文での見方 |
|---|---|---|---|
| ユーザID管理・VPN利用者認証 | 4-1-1-1、4-1-1-2、4-1-1-3、4-1-1-4 | VPN利用者認証・RTX管理者認証 | VPNユーザやRTX管理者をユーザ単位で管理し、削除・無効化・権限変更を説明しやすくする観点です。 |
| RTX管理者ID管理・管理者認証 | 4-1-2-1、4-1-2-2、4-1-2-3、4-1-2-5、4-1-2-6、4-1-2-7、4-1-2-8 | RTX管理者認証 | RTX管理者ログイン、管理アクセス、設定変更権限の対象者を整理する観点です。 |
| パスワード・多要素認証 | 4-1-3-1、4-1-3-3、4-1-3-4、4-1-3-5、4-1-5系、4-1-6系 | VPN利用者認証・RTX管理者認証 | パスワードだけでなくワンタイムパスワードを組み合わせ、パスワードポリシーや漏えい時対応を説明する観点です。 |
| アクセス権管理 | 4-1-7系 | VPN利用者認証・RTX管理者認証 | SingleIDのユーザ・グループ管理、管理権限・一般権限の割り当て、RTX側の管理アクセス制御を組み合わせて、必要最小限の権限を説明する観点です。 |
| ネットワーク境界防護 | 4-5-1-1、4-5-1-2、4-5-1-3、4-5-1-4、4-5-1-7 | RTX設定・運用、RTX管理者認証 | RTX本体の管理パスワード、管理アクセス、フィルタ設定、インターネット経由の管理操作を確認する観点です。 |
| ログ取得・監視 | 4-4-3-1、4-4-3-2、4-4-3-3 | ログ管理 | SingleIDの認証ログとRTX側ログを組み合わせ、認証成功・失敗、不審な試行、設定変更の確認を整理する観点です。 |

## ローカル認証DBとの比較

YAMAHA RTX のローカル認証DBを利用する場合と、SingleIDを併用する場合の主な違いは次のとおりです。

| 観点 | RTXローカル認証DBのみの場合 | SingleIDを併用する場合 |
|---|---|---|
| 認証方式 | パスワードのみの認証になりやすい。 | パスワードにワンタイムパスワードを組み合わせた2要素認証として説明しやすくなる。 |
| ユーザ管理 | VPN利用者やRTX管理者をRTX本体ごとに管理する必要がある。 | SingleID側でユーザを管理し、VPN利用者やRTX管理者の認証を集約しやすくなる。 |
| 退職・異動時の対応 | 機器ごとの設定変更や削除漏れが発生しやすい。 | SingleID側でユーザ停止やグループ変更を行うことで、利用権限の停止を説明しやすくなる。 |
| パスワード漏えい時のリスク | パスワードが漏えいした場合、それだけで認証が成立する可能性がある。 | ワンタイムパスワードを組み合わせることで、パスワード漏えい時のリスク低減を説明しやすくなる。 |
| アカウント共有の抑制 | 共有アカウントを使うと、利用者や管理者の特定が難しくなる。 | ユーザ単位の認証に寄せることで、誰がVPN接続やRTX管理者ログインを行ったかを整理しやすくなる。 |
| 権限管理 | RTX側のローカル設定を機器ごとに管理する必要がある。 | SingleID側のユーザ・グループ管理とRTX側の設定を組み合わせて、権限の整理を説明しやすくなる。 |
| 認証ログ | RTX側ログを中心に確認する必要がある。 | SingleIDの認証ログとRTX側ログを組み合わせて、認証成功・失敗を確認しやすくなる。 |
| 運用上の注意 | ローカル管理者アカウント、緊急時アカウント、管理パスワードの管理が重要になる。 | SingleIDを併用しても、RTX側のローカル管理者アカウント、緊急時運用、ログ保管、申請・承認フローは別途確認が必要。 |

この点で、VPN利用者認証とRTX管理者認証を分けて、全体の認証リスクと運用負荷を比較して判断することが重要です。

## 判断時のポイント

SCS評価制度の観点で確認する場合は、次の観点で整理すると判断しやすくなります。

- VPN利用者認証とRTX管理者認証のどちらにSingleIDを適用するか
- RTX管理者ログインにSingleID認証を利用するか
- SingleID側で管理権限・一般権限をどのユーザ・グループに割り当てるか
- RADIUS障害時の緊急時アクセス手順をどう定めるか
- RTXのローカル管理者アカウントを残す場合、そのパスワード管理と利用条件をどう定めるか
- 管理画面や管理接続をどの範囲に公開するか
- 管理アクセスの接続元ホストをどう制限するか
- VPN通信が暗号化されていること
- 認証基盤との通信経路が適切に制限されていること
- パスワードのみではなくワンタイムパスワードを組み合わせていること
- SingleIDの認証ログとRTX側ログをどの頻度で確認するか
- 設定変更ログや操作証跡をどう保管するか
- 申請・承認フロー、棚卸、台帳管理を顧客側の運用としてどう整備するか
